Chapter 1

Introduction & Overview

What Project RECON is, what it does, and who it is for.

What Is Project RECON?

Project RECON (Rapid Evidence Collection & OSINT Network) is a unified, command-line investigation workbench for macOS. It replaces the need to manually install, configure, and run dozens of separate open-source intelligence (OSINT) and digital forensics tools by orchestrating all of them through a single interface.

Built on top of the WhatsMyName project (a community-maintained database of 4,387+ sites that can be searched for usernames), RECON extends it into a full investigation platform with sessions, chain-of-custody evidence packaging, exposure scoring, relationship investigation, face recognition, and operational security controls.

ℹ️
Built for Investigators
Project RECON is designed for use by licensed private investigators, law enforcement, corporate security teams, and individuals conducting legal OSINT investigations. All outputs are structured for potential use as documented evidence.

  • Runs tools in parallel — live progress dashboard

💑 SPOTLIGHT Module
  • 35+ relationship investigation tools
  • 13 investigation phases

  • Behavioral metadata & geolocation
⚖️ Forensic Standards

One-Command Install (Portable Drive)

If RECON ships on a portable drive, plug it in and run:

bash /Volumes/YOUR_DRIVE/RECON_WorkBench/install.sh

The installer will automatically:

  • Copy Project RECON to ~/RECON_WorkBench
  • Create an isolated Python virtual environment at ~/RECON_WorkBench/.venv
  • Install all 40+ Python OSINT tools via pip
  • Install Homebrew tools (nmap, tshark, tcpdump, exiftool, etc.) if Homebrew is present
  • Add the recon and spotlight commands to your shell PATH
  • Set up a weekly auto-update job (Mondays at 3:00 AM)

Install from GitHub

git clone https://github.com/newbiehackler/project-recon.git
cd project-recon
bash install.sh

Manual / Developer Install

git clone https://github.com/newbiehackler/project-recon.git
cd project-recon
python3 -m venv .venv && source .venv/bin/activate
pip install -e .

# Core OSINT tools
pip install sherlock-project maigret social-analyzer socialscan holehe h8mail nexfil

# SPOTLIGHT tools
pip install ghunt instaloader xeuledoc snscrape pyktok redditsfinder telepathy
pip install osrframework crosslinked linkedin2username daProfiler photon-crawler toutatis

Verifying Installation

recon --version        # Should print version 3.3.0
recon --list-tools     # Shows all tools and their install status
recon inventory        # Full system audit

Uninstalling

bash ~/RECON_WorkBench/uninstall.sh

This removes RECON, the recon/spotlight launchers, auto-update jobs, and shell PATH entries. Your Homebrew and pip tools remain untouched — only RECON-specific components are removed.

Chapter 4

Quick Start

Get your first scan running in under 60 seconds.

First Run

After installation, update the site database to get the latest detection data from Sherlock and Maigret:

recon update

This fetches ~3,500 additional site definitions and caches them locally. After deduplication you will have 4,387+ unique sites to search against.

Basic Scans

Username Search

recon johndoe

RECON auto-detects that johndoe is a username and runs all applicable tools in parallel. It also generates email variants (johndoe@gmail.com, @yahoo.com, etc.) and runs email intelligence tools against those automatically.

Email Search

recon johndoe@gmail.com

Runs: holehe (service registrations), h8mail (breach databases), socialscan (platform availability), theHarvester (domain intelligence).

Phone Number Search

recon +15551234567

Runs: phoneinfoga (carrier/location/search links), ignorant (WhatsApp/Telegram/Signal/Snapchat registration check).

Common Useful Commands

# Deep scan — all tools, 2× timeouts
recon johndoe --deep

# Export as HTML report and open in browser
recon johndoe -o html --open

# Quick scan (top 3 fastest tools, ~60 seconds)
recon johndoe --quick

# Use a predefined template
recon johndoe --template full-osint

# Track in a named session with examiner tag
recon johndoe --session case001 --case-id CASE-2024-001 --examiner "Agent Smith"

# Dry run — preview what WOULD execute without running anything
recon johndoe --dry-run

# Suggest next tools after scan
recon johndoe --discover

# Calculate exposure risk score (1–100)
recon johndoe --score

Understanding the Output

When a scan completes, RECON displays:

  • Live progress bar — shows each tool running, elapsed time, findings per tool
  • Summary table — total unique accounts found, tools completed/failed, duration
  • Findings grouped by category — Social, Dating, Finance, Gaming, etc.
  • Exposure score (if --score) — 1–100 risk rating with category breakdown
  • Next-step suggestions (if --discover) — ranked tool recommendations

Session Tracking

Track multi-run investigations, resume where you left off, and maintain finding history over time.

What Sessions Do

A session is a persistent investigation record stored as a JSON file in ~/.recon/sessions/<name>.json. Sessions allow you to:

  • Resume an investigation across multiple work sessions
  • Aggregate findings from multiple targets under one case
  • Compare findings over time (run on Monday vs. run on Friday)
  • Tag findings with case ID, examiner name, and notes
  • Include the session in evidence bag outputs

Creating and Using Sessions

# Start a new session
recon johndoe --session investigation_001 --examiner "Agent Smith" --case-id CASE-2024-001

# Add notes to a session
recon johndoe --session investigation_001 --notes "Second scan after target changed username"

# Run a different target in the same session
recon johndoe@gmail.com --session investigation_001

# Resume a session (re-runs same targets, appends new findings)
recon --session investigation_001

# Compare current scan against a previous JSON export
recon johndoe --diff previous_scan.json

When you add --evidence-bag ./evidence/ --chain-of-custody to any scan, RECON creates a forensically structured directory containing:

OpSec & Anonymity

Protecting your investigation identity and avoiding detection.
⚠️
Understand Active vs. Passive
Passive tools (flagged passive=True in the catalog) check public records or use indirect methods — they do not directly contact the target. Active tools make direct requests that may be logged by the target's server. Always use --passive mode when you need to avoid leaving traces.

VPN Check

# Refuse to run if no VPN is detected
recon johndoe --vpn-check

# RECON detects VPN by checking for these network interfaces:
# utun0, utun1, utun2 (macOS built-in VPN)
# tun0 (OpenVPN)
# wg0, wg1 (WireGuard)

Tor Routing

# Start Tor first
brew install tor
tor &

# Route all tool requests through Tor
recon johndoe --tor

# Tor + slow-down for realism
recon johndoe --tor --jitter --parallel 3 --retries 2

Proxy Routing

# HTTP proxy (Burp Suite, mitmproxy)
recon johndoe --proxy http://127.0.0.1:8080

# SOCKS5 proxy
recon johndoe --proxy socks5://127.0.0.1:1080

# SPOTLIGHT with proxy
spotlight johndoe --proxy socks5://127.0.0.1:9050

OpSec Check — Pre-Scan Warning

# Warn before running any active (trace-leaving) tools
recon johndoe --opsec-check

RECON will display a list of active tools in the scan and prompt you to confirm before proceeding. Passive tools are identified in the catalog by passive=True and are safe to run without disclosure risk.

Rate Limiting & Jitter

# Max 5 requests per second across all tools
recon johndoe --rate-limit 5

# Random delay between tool launches (0.5–3 seconds)
recon johndoe --jitter

# Combined — hardest to detect pattern
recon johndoe --passive --jitter --rate-limit 2 --parallel 3

Passive-Only Mode

# Run only tools that don't directly contact the target
recon johndoe --passive

# Or use the passive-only template
recon johndoe --template passive-only

Troubleshooting

Solutions to the most common problems.

Command Not Found

⚠️
"recon: command not found"

The recon command is not in your PATH. Solutions:

  1. Re-run the installer: bash ~/RECON_WorkBench/install.sh
  2. Activate the virtual environment: source ~/RECON_WorkBench/.venv/bin/activate
  3. Run directly: python3 -m whatsmyname johndoe
  4. Add to PATH manually: export PATH="$HOME/.local/bin:$PATH"

Tool Not Found / Skipped

If a tool is skipped during a scan, it is not installed. Check with:

recon inventory            # See all tools and install status
recon learn sherlock       # Shows install command for the tool

Install the missing tool with the command shown in recon learn <tool> or recon inventory.

Scan Is Very Slow

  • Use --quick for a fast scan (top 3 tools, ~60 seconds)
  • Reduce timeout: --timeout 8
  • Skip email expansion: --no-email-expand
  • Run Tor with fewer parallel workers: --tor --parallel 3 --timeout 30

Many Timeouts or Errors

  • Increase timeout: --timeout 60
  • Add retries: --retries 3
  • Some sites block automated requests — this is expected behavior
  • Add --raw flag to see which tools are failing and why

Tor Connection Refused

  • Ensure Tor is running: tor & or via Tor Browser
  • Default port is 9050 (not 9150 from Tor Browser)
  • Install required library: pip install aiohttp-socks
  • Test Tor: curl -x socks5://127.0.0.1:9050 https://check.torproject.org

No Results / Zero Findings

  • Run recon update to fetch the latest site database
  • Check the username spelling — RECON searches exact-match by default
  • Try --deep for longer timeouts on slow-responding sites
  • Run recon --list-tools to confirm tools are installed
  • Try --nsfw if the subject may use adult platforms

CompreFace (Face Recognition) Not Starting

  • Ensure Docker Desktop is installed and running
  • Run compreface start and wait ~60 seconds for containers to initialize
  • Check status: compreface status
  • Docker must have at least 2 GB RAM allocated

SSL Errors

  • RECON disables strict SSL verification by default for self-signed certificates
  • If errors persist, try routing through a local proxy: --proxy http://127.0.0.1:8080

Evidence Bag Hash Verification Failure

If --hash-verify reports mismatches, it means files in the evidence bag were modified after collection. This is a critical finding — the integrity of that evidence is compromised. Never modify files inside an evidence bag directory. Always work with copies.

Forensics Glossary

Key terms used throughout Project RECON and digital forensics investigations.
TermDefinition
OSINT Open Source Intelligence — intelligence collected from publicly available sources including social media, websites, public records, and published data.
Chain of Custody A chronological documentation proving the seizure, custody, control, transfer, analysis, and disposition of evidence. Each action is timestamped and logged. Breaks in chain of custody can make evidence inadmissible.
SHA-256 Secure Hash Algorithm 256-bit — a cryptographic hash function that produces a unique 64-character fingerprint of any file. If even one byte changes, the hash changes completely. Used to verify evidence has not been tampered with.
Evidence Bag In digital forensics, a container (directory) that stores all collected artifacts alongside their hash values and collection metadata. Project RECON's EvidenceBag creates MANIFEST.json + CHAIN_OF_CUSTODY.json automatically.
Passive Reconnaissance Gathering information without directly interacting with the target system or person. Examples: WHOIS lookups, DNS queries, social media searches. Leaves no trace on the target's systems.
Active Reconnaissance Gathering information by directly interacting with the target. Examples: port scanning, making HTTP requests to target's servers. May leave logs on the target's systems.
Exposure Score A calculated 1–100 risk rating reflecting how much digital footprint a target has. Higher scores indicate more publicly discoverable information. Calculated by RECON using finding counts across categories.
Enrichment The process of enhancing raw findings with additional context — resolving IPs, performing reverse DNS lookups, adding geolocation data. Run with --enrich.
Correlation Identifying relationships between findings from multiple tools — e.g., the same URL appearing in both WMN and Sherlock results. Run with --correlate.
Jitter Random time delays between automated requests. Mimics human behavior and avoids triggering rate-limit detection. Enabled with --jitter.
Steganography The practice of hiding secret information within ordinary, non-secret data — most commonly hiding text or files inside image files.
EXIF Exchangeable Image File Format — metadata embedded in digital photos including camera model, settings, date/time, and critically: GPS coordinates where the photo was taken.
GAIA ID Google Account ID — a unique numeric identifier for a Google account. Can be used to cross-reference Google Maps reviews, YouTube channel ownership, and other Google services.
Tor The Onion Router — a network that anonymizes internet traffic by routing it through multiple encrypted relay nodes. Project RECON routes all tool requests through Tor when --tor is used.
SOCKS5 A network proxy protocol that routes TCP connections through a proxy server. Used by Tor (socks5://127.0.0.1:9050) and supported by all RECON tools via --proxy.
Data Breach An unauthorized exposure of private information — passwords, emails, personal data — from a hacked service. Tools like h8mail and holehe check whether a target's email appears in known breaches.
WHOIS A query protocol that returns registration information for a domain or IP address: registrant name, organization, registrar, creation date, expiry date, and nameservers.
MX Record Mail Exchange DNS record — specifies which mail servers handle email for a domain. Useful for identifying email providers and understanding infrastructure.
SPF / DKIM Sender Policy Framework / DomainKeys Identified Mail — email authentication records stored as DNS TXT records. Reveal which services are authorized to send email for a domain.
Plugin A Python file placed in ~/.recon/plugins/ that adds new tools or capabilities to Project RECON. Automatically loaded on startup.
LaunchAgent (launchd) macOS background job scheduler. Project RECON installs a LaunchAgent (dev.recon.autoupdate.plist) that runs the auto-update script every Monday at 3:00 AM.
